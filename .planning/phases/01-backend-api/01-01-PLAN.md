  <action>
    Create server/server.js with the following implementation. This is a SINGLE FILE with all routes inline -- no controllers, no routes directory. The student must explain this code during grading, so every line must be intentional and clear.

    ```javascript
    import express from 'express'
    import cors from 'cors'
    import { JSONFilePreset } from 'lowdb/node'

    const app = express()
    const PORT = 3000

    // Middleware: CORS first (before routes), then JSON body parser
    app.use(cors())
    app.use(express.json())

    // Initialize LowDB with seed data from db.json
    // If db.json doesn't exist, JSONFilePreset creates it with defaultData
    const defaultData = { products: [], cart: [] }
    const db = await JSONFilePreset('db.json', defaultData)

    // GET /products - return all products
    app.get('/products', (req, res) => {
      res.json(db.data.products)
    })

    // GET /cart - return all cart items
    app.get('/cart', (req, res) => {
      res.json(db.data.cart)
    })

    // POST /cart - add item to cart with validation
    app.post('/cart', async (req, res) => {
      const { productId, name, price, quantity } = req.body

      if (!productId || !name || price == null || !quantity) {
        return res.status(400).json({ error: 'Missing required fields: productId, name, price, quantity' })
      }

      const newItem = { productId, name, price, quantity }
      db.data.cart.push(newItem)
      await db.write()

      res.status(201).json(newItem)
    })

    app.listen(PORT, () => {
      console.log(`Server running on http://localhost:${PORT}`)
    })
    ```

    Key implementation details (the student must understand these):
    - `app.use(cors())` MUST be the first middleware -- before express.json() and before any routes. If CORS comes after routes, preflight OPTIONS requests fail (Research Pitfall 1).
    - `express.json()` parses JSON request bodies -- without it, req.body is undefined on POST.
    - `JSONFilePreset('db.json', defaultData)` reads existing db.json OR creates it with defaultData. Since we ship db.json with 11 products, the seed data loads on first run.
    - `await db.write()` after cart modifications persists to db.json via steno (atomic writes). Without this, cart data is lost on server restart (Research Pitfall 2).
    - Top-level `await` works because package.json has `"type": "module"` (ESM). CommonJS would fail with ERR_REQUIRE_ESM (Research Pitfall 3).
    - POST /cart validates all 4 required fields and returns 400 if any are missing. Returns 201 with the saved item on success (Research Pattern 3).
    - `price == null` catches both null and undefined. Using `===` would miss undefined.

## Trust Boundaries

| Client (Ionic app) → Express API | Untrusted input crosses here via HTTP requests |
| Express API → db.json file | Trusted local file write (same machine) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | POST /cart body | mitigate | Validate all 4 required fields (productId, name, price, quantity) before storing. Return 400 with descriptive error if any field is missing. Reject requests with null/undefined values. |
| T-01-02 | Information Disclosure | CORS policy | accept | cors() with no origin restriction is acceptable for localhost lab exam development. Production would restrict origin. Noted as accepted risk for lab exam scope. |
| T-01-03 | Tampering | db.json concurrent writes | mitigate | LowDB uses steno for atomic file writes, preventing corruption from concurrent requests. No additional code needed -- handled by library. |
| T-01-04 | Tampering | POST /cart price manipulation | accept | Client can submit any price value. No authentication or authorization exists -- this is a lab exam project. Accepting as scoped out. A production system would require auth and server-side price validation against the products catalog. |
| T-01-05 | Denial of Service | Unbounded cart array | accept | No rate limiting or cart size limits. Acceptable for lab exam localhost demo. Production would add rate limiting and size constraints. |
</threat_model>
